# Alembic migrations

Wired for **async** engines + **batch mode** (SQLite) + portable **`Money`** rendering. The DB URL
comes from `TT_LEDGER_DATABASE_URL` (default `sqlite+aiosqlite:///ledger.db`), so the *same* revision
set runs on SQLite and Postgres.

## Usage

```bash
# autogenerate a revision after changing models (tt_ledger/schema/models.py)
TT_LEDGER_DATABASE_URL="sqlite+aiosqlite:///dev.db" \
  alembic revision --autogenerate -m "describe change"

# apply
TT_LEDGER_DATABASE_URL="sqlite+aiosqlite:///dev.db"      alembic upgrade head     # SQLite
TT_LEDGER_DATABASE_URL="postgresql+asyncpg://…/ledger"   alembic upgrade head     # Postgres
```

Run from the repo root (`alembic.ini` sets `prepend_sys_path = .`).

## How portability is preserved

- `env.py` sets `render_as_batch=True` → SQLite `ALTER TABLE` works (batch rebuilds the table).
- `env.py`'s `render_item` renders every `Money` column as `Money(scale=6)` (not the dialect-specific
  impl), so a migration autogenerated against SQLite still creates `NUMERIC(18,6)` on Postgres at apply
  time. The generated migration imports `from tt_ledger.money import Money`.
- Enum columns are `String`; foreign keys avoid the orders↔trade_groups cycle (see `models.py`), so
  there is no `ALTER ADD CONSTRAINT` that SQLite cannot express.

Review every autogenerated revision before committing (autogenerate is a starting point, not gospel).
